# 1.20.1 Forge -> 1.21.1 NeoForge Mek Port Notes

- Scope: only the Mekanism integration pipeline and the two custom matter value additions that do not already exist upstream in 1.21.1, namely `c:gems/fluorite` and `minecraft:water`.
- Background: the other 1.20.1 matter value recipes were backported from 1.21.1, so they are intentionally not re-imported here.

## Loader and platform differences

- Event bus: 1.20.1 used Forge event classes under `net.minecraftforge.*`; 1.21.1 uses NeoForge equivalents under `net.neoforged.*`.
- Reload hook: `AddReloadListenerEvent` moved to `net.neoforged.neoforge.event.AddReloadListenerEvent`.
- Fluids: `FluidStack` moved from `net.minecraftforge.fluids.FluidStack` to `net.neoforged.neoforge.fluids.FluidStack`.
- Mod lookup: `ModList` moved from `net.minecraftforge.fml.ModList` to `net.neoforged.fml.ModList`.

## Replication differences

- Custom matter defaults: 1.21.1 `ReplicationCalculation.DEFAULT_MATTER_RECIPE` stores `RecipeHolder<MatterValueRecipe>` entries, not bare recipes.
- Export table: 1.21.1 `ReplicationCalculation.DEFAULT_MATTER_COMPOUND` is keyed by `Item`, not by string item id.
- Sync tag serialization: 1.21.1 `MatterCompound.serializeNBT` requires `RegistryAccess`.
- Mixin target: 1.20.1 intercepted `calculateRecipes()` with no args; 1.21.1 intercepts `calculateRecipes(RegistryAccess)`.
- Startup timing: on 1.21.1 the first `calculateRecipes(RegistryAccess)` call can happen before a live server exists, so the port adds a server-start recalculation hook and skips the premature client-side invocation.
- Recipe payload shape: 1.21.1 `MatterValueRecipe` JSON entries use `amount`, while the custom node-value JSON still uses `value`.
- Datapack path: 1.21.1 recipe JSONs live under `data/<namespace>/recipe/...` instead of the older `recipes/...` path.
- Tag namespace: upstream 1.21.1 Replication uses `c:` tags instead of `forge:` tags for shared matter value recipes.

## Mekanism differences

- Recipe types: several classes that were imported from `mekanism.api.recipes.chemical.*` in 1.20.1 are available directly under `mekanism.api.recipes.*` in 1.21.1.
- Compressing detection: 1.20.1 used `ItemStackGasToItemStackRecipe` plus a recipe type id check; 1.21.1 no longer exposes that subclass, so the port checks `MekanismRecipeTypes.TYPE_COMPRESSING`.
- Chemical dissolution outputs: 1.20.1 treated the JEI output wrapper as an object containing `chemicalStack`; 1.21.1 exposes `ChemicalStack` directly in `outputDefinition`.
- Rotary naming: the port uses fluid/chemical wording to match the 1.21.1 API (`hasFluidToChemical`, `hasChemicalToFluid`).
